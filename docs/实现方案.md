# AI 商品打标服务实现方案

## 一、项目目标

为二手交易平台构建一套自动化商品图片标签生成服务，实现：

- 输入：商品主图（单张或多张）
- 输出：结构化标签（类目、颜色、材质、风格、成色等二手交易关键字段）
- 目标：90% 以上标准图由本地模型直接处理，疑难图走 DeepSeek-VL API + 本地工具兜底

## 二、整体实现路径

```
数据准备（200 张）
    ↓
DeepSeek API 打标 + 人工审核
    ↓
构建微调数据集
    ↓
Qwen2-VL-7B INT8 本地微调
    ↓
服务部署（vLLM + FastAPI 工具服务）
    ↓
灰度上线 → 数据回流 → 持续迭代
```

## 三、训练数据获取与构建

### 3.1 数据来源

| 来源 | 数量 | 说明 |
|------|------|------|
| 平台存量商品图片 | 100 张 | 覆盖平台真实类目分布，优先选择白底、清晰、单商品主图 |
| 开源二手商品数据集 | 100 张 | `Nilanjan-2002/fashion-second-hand-front-only-rgb` 取 30 张；`NingLab/MMECInstruct` 中 `category_classification` 子集取 70 张，补充 3C / 家居等非服饰场景 |
| 合计 | 200 张 | 作为首期微调种子数据 |

脚本 [`scripts/prepare_hf_datasets.py`](file:///d:/ai-tagging-flow/scripts/prepare_hf_datasets.py) 会自动完成下载、GLM-4V 中文标准化打标，并生成 `image_label_mapping.json` 保存图片与标签的对应关系。

### 3.2 数据收集规范

1. **存量商品图片筛选标准**
   - 图片清晰，主体可辨识
   - 覆盖平台核心类目：服饰、3C、家居、美妆、鞋包等
   - 单张图单主体优先，复杂场景图占比不超过 30%
   - 图片尺寸不低于 512×512

2. **开源数据采集**
   - 优先选择带类目标注的公开数据集
   - 避免采集人脸、隐私信息、侵权内容
   - 记录每张图片的来源 URL 或数据集名称，便于合规审查

3. **数据去重与清洗**
   - 使用 perceptual hash 去重
   - 删除模糊、过度压缩、含明显水印的图片
   - 统一为 JPG/PNG 格式

### 3.3 数据标注流程

**第一步：DeepSeek API 自动打标**

调用 DeepSeek-VL API，构造如下提示词：

```text
你是一位专业的二手商品图片标注员。请仔细观察图片，输出以下结构化特征标签：

- 类目：
- 颜色：
- 材质/面料：
- 款式特征：
- 成色：

要求：
1. 标签简洁、标准化，便于搜索和推荐
2. 无法判断的字段填写"未知"
3. 优先使用平台常用词
```

返回格式统一为 JSON，便于后续解析。

**第二步：人工审核**

- 由 1-2 名业务人员逐条核对 DeepSeek 输出标签
- 审核重点：类目是否正确、颜色/材质是否准确、是否存在幻觉标签
- 对错误标签直接修正，对不确定项标注为"待确认"
- 审核通过率计入质量指标

**第三步：数据集格式化**

按 LLaMA Factory 训练格式整理为对话式样本：

```json
{
  "images": ["path/to/image.jpg"],
  "messages": [
    {
      "role": "user",
      "content": "请为这张二手商品图片生成标准化标签，包含类目、颜色、材质/面料、款式特征、成色。"
    },
    {
      "role": "assistant",
      "content": "类目：连衣裙\n颜色：黑色\n材质/面料：聚酯纤维\n款式特征：简约通勤\n成色：九成新"
    }
  ]
}
```

### 3.4 如何利用数据集中的已有标签

两个开源数据集的字段不尽相同，处理方式也不同：

**`fashion-second-hand-front-only-rgb`**：该数据集已包含 `category`、`type`、`colors`、`material`、`condition`、`cut`、`pattern` 等结构化字段，**可以直接作为初始标签**，无需让 DeepSeek 从零看图打标。推荐做法：

1. 读取已有字段，映射到目标格式（类目 / 颜色 / 材质/面料 / 款式特征 / 成色）
2. 将已有标签作为参考信息传入 DeepSeek 提示词
3. DeepSeek 只做三件事：**标准化措辞**、**修正明显错误**、**补全缺失字段**
4. 输出仍由人工审核，审核重点是 DeepSeek 是否改坏了已有正确标签

**`MMECInstruct`**：该数据集是指令微调格式，包含 `category_classification`、`sentiment_analysis` 等多种任务。与商品打标直接相关的是 `category_classification` 子集，可利用其图片和标题辅助生成标签；其他任务（如情感分析、推荐）与目标场景关联较弱，不建议直接混入训练集。

**是否可以完全替代 DeepSeek API？**

- 对于 `fashion-second-hand-front-only-rgb`，如果已有标签质量高，可以先用已有标签生成训练样本，**只在需要标准化或补全时调用 DeepSeek**，能显著降低 API 成本
- 对于平台存量图和无标签图，仍需 DeepSeek 打标
- 建议保留 `raw_labels` 和 `deepseek_labels` 两份记录，便于人工审核时对比

已提供脚本 [`scripts/prepare_hf_datasets.py`](file:///d:/ai-tagging-flow/scripts/prepare_hf_datasets.py) 完成下载、字段映射、GLM-4V 标准化和训练格式导出。

## 四、模型微调方案

### 4.1 基础模型

- 模型：Qwen2-VL-7B-Instruct
- 微调方式：QLoRA（4-bit 量化 + LoRA）
- 训练框架：LLaMA Factory
- 推理部署：vLLM INT4 AWQ 量化

### 4.2 训练配置

```yaml
model_name_or_path: Qwen/Qwen2-VL-7B-Instruct
finetuning_type: lora
quantization_bit: 4
lora_rank: 64
lora_alpha: 128
lora_target: all
learning_rate: 1.0e-4
num_train_epochs: 3
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
cutoff_len: 2048
max_samples: 1000
```

### 4.3 训练目标

- 训练任务：从商品图片中抽取标准化标签
- 不训练工具调用能力，工具调用逻辑由 DeepSeek API 分支承担
- 每轮训练后通过预留验证集评估标签准确率

## 五、服务架构实现

### 5.1 整体架构

沿用《大作页方案.md》中的分层设计：

| 层级 | 组件 | 职责 |
|------|------|------|
| 入口层 | 业务中台 API | 接收图片、调度流程、返回标签 |
| 本地模型层 | Qwen2-VL-7B INT8 (vLLM) | 处理标准图，直接输出标签 |
| 决策层 | DeepSeek-VL API | 处理疑难图，决定调用哪些本地工具 |
| 工具层 | RMBG-2.0 / YOLOv8-Nano | 抠图、单品裁切等本地预处理 |

### 5.2 核心流程伪代码

```python
async def tag_image(image_url: str):
    # 1. 图片压缩预处理
    image = await preprocess(image_url)

    # 2. 本地模型初筛
    result, confidence = await local_model.predict(image)

    # 3. 置信度足够且单主体白底图，直接返回
    if confidence >= 0.75 and is_clean_background(image):
        return standardize_tags(result)

    # 4. 否则进入 DeepSeek 疑难分支
    processed_images = await deepseek_tool_branch(image)

    # 5. DeepSeek 生成图文描述后回传本地模型标准化
    description = await deepseek_describe(processed_images)
    final_tags = await local_model.standardize(description)

    return final_tags
```

### 5.3 本地工具接口

两个 FastAPI 内网服务：

| 服务 | 路径 | 输入 | 输出 |
|------|------|------|------|
| 抠图服务 | `/remove_bg` | 图片文件/URL | 白底商品图 |
| 检测裁切服务 | `/detect_crop` | 图片文件/URL | 子图列表 + 坐标 |

所有工具限制最大边长 768，超时自动降级为原图识别。

## 六、部署方案（RTX 4060 8G）

### 6.1 常驻服务

- Qwen2-VL-7B INT8：约 6GB 显存
- 工具服务（按需加载）：抠图、检测共用剩余显存

### 6.2 并发控制

- 本地识别并发：≤4
- 工具处理并发：≤2
- 使用显存隔离或分时调度避免 OOM

### 6.3 技术栈

| 模块 | 技术 |
|------|------|
| 模型推理 | vLLM + Transformers |
| 微调 | LLaMA Factory |
| 工具服务 | FastAPI + Docker |
| 抠图 | RMBG-2-Studio |
| 目标检测 | YOLOv8-Nano |
| 图片处理 | Pillow / OpenCV |
| 数据管理 | JSONL / 本地文件系统 |

## 八、质量与风险控制

### 8.1 关键指标

| 指标 | 目标 |
|------|------|
| 本地模型处理比例 | ≥90% |
| 本地模型 Top-1 类目准确率 | ≥85% |
| 人工审核修正率 | ≤15% |
| 单张平均耗时 | ≤1.5s（本地） |
| DeepSeek 分支平均耗时 | ≤5s |

### 8.2 风险与应对

| 风险 | 应对措施 |
|------|----------|
| 200 张数据量不足 | 上线后持续回流优质清洗图，按月扩充训练集 |
| DeepSeek API 打标成本高 | 仅用于疑难图和训练数据生成，90% 流量走本地 |
| 模型幻觉 | 输出标签限制在固定字段，人工审核拦截 |
| 显存不足 | 限制输入尺寸、控制并发、工具按需加载 |
| 开源数据合规问题 | 记录来源、去除人脸隐私、避免商用侵权数据集 |
